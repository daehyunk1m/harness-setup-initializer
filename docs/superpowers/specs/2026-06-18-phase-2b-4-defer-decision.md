## 0. 한 줄 요약

2b-4 후보 두 검사(**feature→PRD 교차**, **PRD→검증 역방향 "미검증 명세"**)를 **지금 짓지 않는다.** 보수성 게이트("입증 못하면 뺀다")를 통과하지 못했다 — 게이트로 쓰려던 `passes` 신호가 약하고, 역방향은 기존 커버리지 추적과 중복되며, 교차는 README "PRD 없음=온디맨드 정상" 정책과 충돌한다. **코드 변경 0.** 4사분면 프레이밍(§2)과 재검토 트리거(§5)·미래 시작점(§6)을 박제해 다음 세션이 0에서 시작하지 않게 한다.

이는 brainstorming의 정당한 종착점이다 — "검증된 설계 = 짓지 않는다는 결정." 2b-3에서 이슈 #15의 핵심은 이미 닫혔고(열린 이슈 0), 2b-4는 처음부터 "보수성 입증 후" **조건부**였다. 이연은 깨진 상태를 남기지 않는다.

## 1. 검토 대상 (무엇을 짓는다는 것이었나)

이슈 #15 추적 파이프라인(대화→원장→증류→정적추적)의 마지막 조각. harness-check ⑩ "PRD 위생"엔 이미 마커 위생(Inc1)·내용 위생(Inc2)이 있고, 2b-4는 feature↔PRD **관계**(존재/검증 정합)를 검사하려 했다:

- **forward 교차**: `feature_list.json`에 있으나 바인딩 PRD 없는 feature 표면화 (`ids − seen` — `prd_marker_hygiene`이 이미 두 집합을 계산하므로 데이터 비용 ≈ 0).
- **reverse "미검증 명세"**: PRD는 있으나 검증(`passes:true` 또는 `@feature` E2E)이 없는 것.

게이트(프로젝트 일관 테마, codex): **둘 다 "보수성 입증 못하면 뺀다."**

## 2. 분석 — 4사분면 추적성 프레이밍 (박제)

각 feature를 두 축으로 본다 — **PRD 있음?** × **검증됨?**(`passes:true` 또는 `@feature` E2E 존재):

| | PRD 있음 | PRD 없음 |
|---|---|---|
| **검증됨** | ① 이상적 — 무경고 | **③ 검증됐는데 PRD 없음** ← forward 고신뢰 |
| **미검증** | **② PRD 있는데 검증 안 됨** ← reverse | ④ 온디맨드 정상 — **무경고** |

**핵심 통찰**: 블랭킷 "PRD 없는 feature 경고"(③+④)는 ④를 포함해 노이즈 폭탄(신선한 프로젝트는 전부 ④). **검증 신호(`passes`)로 게이팅하면** ④가 자동 침묵하고 ②·③만 남아, 블랭킷 교차보다 안전하다. 이것이 핸드오프가 암시한 "고신뢰(`passes:true`인데 PRD 없음)만" 옵션의 일반화다.

**그러나 이 프레이밍의 토대(`passes`)가 약하다 — 그래서 이연한다(§3·§4).**

## 3. 멀티모델 자문 종합 (codex 결함/노이즈 · gemini 대안/운영)

### 합의 (둘 다 일치)
- **(c) 지금 경고 검사로 짓지 말 것 (이연).** 보수성 게이트 통과 실패.
- **per-item 경고는 최악** — 양치기 소년/경고 피로. 마커·빈섹션(명확한 위반)과 달리 "검증됐는데 PRD 없음"은 *스타일 권장*에 가깝다.
- **정책 모순**: README가 "PRD 없어도 정상"이라 명시하는데 그걸 경고하면 자기모순.
- 반드시 넣어야 하면 **WARN이 아니라 INFO 요약줄 한 줄**(exit 0, per-item 목록 미출력).

### 상충 (생산적 불일치)
- **`passes` 게이트 견고성** — codex: *약하다.* `passes`는 "어떤 형태로든 확인됨"(품질 보증 아닌 **진행 상태** 신호)이라 ③에 오탐(임시 수동확인·핫픽스·내부 feature·`steps` 수행만으로 통과·관례적 true). gemini 동의(핫픽스·UI 텍스트 변경도 feature_list에 기록 가능).
- **forward(③) vs reverse(②) 어느 게 더 가치 있나** — **정면 충돌**:
  - codex: ③이 낫다. ②는 기존 `@feature` E2E 커버리지/INTENT_BACKLOG와 **중복**, "스펙 있고 아직 구현 전"(정상)과 "구현됐는데 검증 누락"(actionable)이 섞여 구분 불가.
  - gemini: ②가 낫다. PRD 존재 = 누군가 시간 들인 중요 기능 → 검증 누락 통지가 actionable.
  - → **두 모델이 서로 반대 방향을 "더 나은 쪽"으로 지목 = 어느 쪽도 명확히 우월하지 않다는 신호.**

## 4. 결정 — 이연 (근거)

제 4사분면 프레이밍은 블랭킷 교차보다 안전하지만, 자문이 그 토대가 약함을 정확히 짚었다. 이연 근거:

1. **`passes`는 "PRD 필요성"을 증명하지 못한다** — 게이트로 ④ 노이즈는 막지만, ②③ 경고의 *근거*로는 의미가 섞여 있다(진행 상태 ≠ 품질 보증).
2. **②는 기존 추적과 중복, ③은 README 정책과 충돌** — 둘 다 좋은 검사의 3원칙(결정적·이견없음·즉시조치가능) 중 최소 하나를 어긴다.
3. **codex↔gemini가 forward/reverse를 정반대로 평가** — 가치 *방향*조차 합의 안 됨 → "입증" 실패.
4. **pull 수요 없음** — 사용자 본인 입장이 "가치 불확실", 도그푸딩에서 이 부재로 인한 구체적 마찰 미관찰. 가장 깨끗한 시스템은 "사용자가 실제 요구하는 것만 있을 때"(gemini).

## 5. 재검토 트리거 (언제 다시 보나)

다음 중 **하나라도** 충족하면 2b-4를 재검토한다(그 전엔 짓지 않는다):

- **구체적 pull 수요**: "PRD 없이 검증된 feature가 누적돼 기술 부채/온보딩 마찰을 일으킨다"는 데이터/사례가 도그푸딩 또는 사용자 보고에서 나옴.
- **`passes`의 강화**: feature_list가 검증 강도를 구분(예: `passes` + 검증 출처/강 검증 플래그)하게 진화 → ③의 게이트가 견고해짐. (단 신규 필드는 MINOR 이상 비용 — 별도 결정.)
- **README 정책 변경**: "검증된 feature는 PRD 권장/필수"로 규칙이 바뀌면 ③ 경고가 정책과 정합.
- **forward derive 운영 데이터**: 2b-2 forward 커버리지 derive(intent-distill §4.1)의 실사용 데이터가 역방향 수요를 입증(TODO-108 진입 조건).

## 6. 미래 시작점 (박제 — 그대로 쓰면 안 되는 이유 포함)

다시 착수할 때의 출발점과 함정:

- **검증-게이팅 프레이밍은 살아있다**(§2). 블랭킷 교차는 ④ 때문에 무조건 노이즈 — 검증 게이트가 필수 전제.
- **단, per-item WARN은 첫 형태로 부적합.** 최소 형태는 **INFO 요약줄**(`ℹ️ PRD coverage: verified-without-PRD=N, PRD-without-verification=M`, exit 0, per-item 미출력). codex/gemini 공통 권고.
- **reverse(②)는 PRD 위생이 아니라 coverage/backlog 파생 지표**(codex). 짓는다면 ⑩이 아니라 intent-distill/INTENT_BACKLOG 쪽이 자연스러울 수 있다 — 중복 회피.
- **harness-check ⑩ 데이터는 이미 준비됨**: `prd_marker_hygiene`의 `seen[]`(PRD-바인딩 id)·`ids`(feature_list 전체 id). 재사용하면 forward 교차는 추가 파싱 0.
- **`passes`를 강 게이트로 신뢰하지 말 것** — §3 codex 오탐 목록을 먼저 무력화할 추가 전제(priority/ship 상태/검증 강도)가 안정적으로 확보된 뒤에만 WARN으로 승격.
