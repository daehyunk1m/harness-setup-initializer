# 버전 관리 정책

> 이 스킬은 [Semantic Versioning 2.0.0](https://semver.org/)을 따른다.
> 버전 형식: **MAJOR.MINOR.PATCH** (예: `1.2.3`)

---

## 1. Public API 선언

semver는 "Public API를 선언하라"고 요구한다. 이 스킬의 Public API는 다음 **4개 계약**이다:

| 계약 | 파일 | 소비자 | 안정성 |
|------|------|--------|--------|
| **프로필 스키마** | `.harness-profile.json` | harness-scaffold | 높음 (21개 플레이스홀더의 원천) |
| **매니페스트 스키마** | `.harness-manifest.json` | 업그레이드 시스템 | 높음 (마이그레이션 키) |
| **프리셋 스키마** | `presets/*.json` | 프리셋 매칭 로직 | 보통 |
| **생성 파일 구조** | 템플릿 → 생성된 18개 파일 | AI 에이전트, 사용자 | 보통 |

Public API에 포함되지 **않는** 것:
- SKILL.md 내부 로직 (소크라테스 문답 순서, 스캔 알고리즘)
- 트래킹 문서 (.tracking/*)
- 참조 문서 (references/*)
- 스킬 호출 인터페이스 (frontmatter, hooks)

## 2. 버전 범프 기준

### MAJOR (X.0.0) — 기존 하네스와 호환 불가

**원칙**: 기존에 생성된 하네스가 **정상 동작하지 못하게** 되는 변경.

**구체적 트리거**:

| 변경 | 예시 | 왜 MAJOR인가 |
|------|------|-------------|
| 프로필 필수 필드 제거/이름 변경 | `srcRoot` → `sourceRoot` | 21개 플레이스홀더 중 하나의 원천이 사라져 scaffold가 실패한다 |
| 프로필 필드 타입 변경 | `pathAlias: string` → `object` | 템플릿 치환 로직이 깨진다 |
| 플레이스홀더 제거 | `{{VALIDATE_COMMAND}}` 삭제 | session-routine.md에서 검증 명령이 사라진다 |
| 파일 카테고리 변경 | `CLAUDE.md`를 custom → managed로 | 업그레이드 시 사용자 편집이 덮어써진다 |
| 생성 파일 규칙의 의미 반전 | TDD 사이클에서 RED 단계 제거 | AI 에이전트가 테스트 없이 구현을 완료한다 |
| 매니페스트 version 해석 변경 | 마이그레이션 키 체계 변경 | 기존 업그레이드 경로가 작동하지 않는다 |
| 사용 중인 프리셋 삭제 | `react-next.json` 제거 | 매니페스트에 저장된 프리셋 참조가 깨진다 |

**MAJOR 범프 시 필수 작업**:
1. 마이그레이션 레지스트리에 `M-{이전}-to-{새}` 항목 추가
2. CHANGELOG에 **Breaking Changes** 섹션 작성
3. 업그레이드 가이드 작성 (무엇을 왜 바꿨는지, 사용자가 할 일)

### MINOR (X.Y.0) — 기존 하네스 호환, 새 기능 추가

**원칙**: 기존 하네스가 **그대로 동작**하면서 새 기능이 추가되는 변경.

**구체적 트리거**:

| 변경 | 예시 | 왜 MINOR인가 |
|------|------|-------------|
| 프로필에 선택 필드 추가 | `extras.monitoring: {}` | 기존 프로필에 없으면 해당 섹션이 생략될 뿐 깨지지 않는다 |
| 기본값 있는 새 플레이스홀더 추가 | `{{NEW_FIELD}}` (기본값 `"default"`) | 기존 프로필은 기본값으로 치환된다 |
| 새 프리셋 추가 | `vue-nuxt.json` | 기존 프리셋 매칭에 영향 없다 |
| 새 에이전트 파일 추가 | `agents/qa-engineer.md` | 기존 에이전트에 영향 없다 |
| 템플릿에 새 규칙/섹션 추가 | session-routine.md에 "Plan 모드 통합" | 기존 TDD 사이클은 그대로, 새 기능이 추가된다 |
| 새 파일 카테고리 추가 | data 외에 `derived` 카테고리 | 기존 카테고리 동작은 변하지 않는다 |
| 마이그레이션 레지스트리 확장 | 새 `M-X-to-Y` 항목 | 기존 마이그레이션에 영향 없다 |

### PATCH (X.Y.Z) — 버그 수정, 내부 개선

**원칙**: Public API에 영향 없는 **내부 수정**.

**구체적 트리거**:

| 변경 | 예시 | 왜 PATCH인가 |
|------|------|-------------|
| 템플릿 오타/문법 수정 | session-routine.md 오타 교정 | AI가 따르는 규칙의 의미가 변하지 않는다 |
| 기존 규칙 명확화 | "간소화 조건"을 더 구체적으로 | 의미가 같고, 해석이 명확해질 뿐이다 |
| SKILL.md 내부 로직 수정 | 소크라테스 문답 순서 최적화 | 생성 결과물이 동일하다 |
| 프리셋 설명 텍스트 수정 | `displayName` 변경 | 매칭 로직에 영향 없다 |
| 에러 메시지 개선 | "프로필이 없습니다" 문구 변경 | 사용자 경험만 개선된다 |
| 권한/설정 버그 수정 | `allowed-tools` 누락 추가 (Issue #2) | 기능이 원래 의도대로 동작하게 된다 |
| 디스커버리/인프라 수정 | 심볼릭 링크 경로 수정 (Issue #3) | 스킬 로딩 문제, 생성물에 영향 없다 |

## 3. 판단이 어려운 경우

### 금지 규칙 추가 — MINOR인가 PATCH인가?

**MINOR**. 금지 규칙은 AI 에이전트의 행동을 제약하는 **새 기능**이다. 기존 동작을 깨뜨리진 않지만, 에이전트가 이전에는 허용된 행동을 더 이상 하지 못한다. 이것은 "기능 추가"에 해당한다.

### 에이전트 프롬프트 최적화 — MINOR인가 PATCH인가?

같은 행동을 더 잘 유도하는 프롬프트 개선 = **PATCH**.
에이전트의 행동 범위가 변하는 프롬프트 변경 = **MINOR**.

### 검증 체크리스트 항목 추가 — MINOR인가 PATCH인가?

Phase 3 검증에 항목을 추가하면 이전에 통과하던 것이 실패할 수 있다. 하지만 검증은 스캐폴딩 시에만 실행되므로 **기존 하네스에 영향이 없다** → **MINOR**.

### 기존 규칙의 행동 변경 — MINOR인가 MAJOR인가?

규칙의 **의미**가 바뀌면 MAJOR. 예: "Implementer 3회 실패 → Debugger"를 "Implementer 1회 실패 → Debugger"로 바꾸면 에스컬레이션 동작이 달라진다. 기존 하네스의 에이전트가 예상과 다르게 행동한다 → **MAJOR**.

## 4. 버전 단일화

### 문제: 현재 두 개의 버전이 공존

| 이름 | 위치 | 현재 값 | 용도 |
|------|------|---------|------|
| 스킬 버전 | `project-context.md` | v5.2 | 스킬 진화 이력 |
| 하네스 스키마 버전 | `.harness-profile.json`, `.harness-manifest.json` | "3.3" | 업그레이드 경로 결정 |

두 버전의 공존은 혼란을 일으킨다. 어떤 것을 올려야 하는지 불명확하고, 둘이 같은 변경에 대해 다른 판단을 내릴 수 있다.

### 해결: 하나의 semver로 통합

semver 도입 시점부터 **하나의 버전**만 사용한다:

- `project-context.md` 버전 히스토리 → semver
- `.harness-profile.json` version 필드 → 같은 semver
- `.harness-manifest.json` harness.version → 같은 semver
- `.tracking/CHANGELOG.md` 릴리스 태그 → 같은 semver
- `git tag` → 같은 semver

### semver 시작 버전: 1.0.0

semver FAQ: *"소프트웨어가 프로덕션에서 사용 중이라면 이미 1.0.0이어야 한다."*

이 스킬은 실제 프로젝트에서 사용 중이고, 4개 계약(프로필/매니페스트/프리셋/생성파일)이 안정적이다. **현재 상태를 1.0.0으로 선언**한다.

기존 매니페스트의 "3.3"은 pre-semver 레거시로 취급하여, 부트스트랩 마이그레이션으로 1.0.0에 편입한다.

## 5. 버전 관리 프로세스

### 5.1 변경 시 버전 판단 플로우

```
변경이 있는가?
  ├─ 기존 하네스가 깨지는가? (생성물이 오동작/생성 실패)
  │   └─ YES → MAJOR
  ├─ 새 기능이 추가되는가? (새 규칙, 새 필드, 새 프리셋, 새 에이전트)
  │   └─ YES → MINOR
  └─ 위에 해당 없음 (버그 수정, 오타, 내부 개선)
      └─ PATCH
```

### 5.2 `/gc` 커밋 시 버전 판단

모든 커밋에 버전을 올리지 않는다. **릴리스 단위**로 올린다.

하나의 릴리스는 여러 커밋을 포함할 수 있다. 릴리스 시점에 포함된 변경 중 **가장 높은 수준**이 버전을 결정한다:

```
커밋 1: PATCH (오타 수정)
커밋 2: MINOR (새 에이전트 추가)
커밋 3: PATCH (에러 메시지 개선)
→ 릴리스 = MINOR (가장 높은 수준)
```

### 5.3 릴리스 프로세스

1. **작업 완료** — 논리적 단위의 변경이 모두 커밋됨
2. **버전 판단** — 위 플로우에 따라 MAJOR/MINOR/PATCH 결정
3. **버전 범프** — 다음 파일의 버전 번호를 동시에 업데이트:
   - `references/project-context.md` § 4 버전 히스토리에 항목 추가
   - `.tracking/CHANGELOG.md` 릴리스 헤더
   - SKILL.md 프로필 출력 스키마의 `"version"` 필드
   - harness-scaffold/SKILL.md 프로필 입력 스키마, 매니페스트 스키마의 `"version"` 필드
4. **태그** — `git tag v{X.Y.Z}`
5. **Push** — `git push && git push --tags`

MAJOR 범프 시 추가:
6. 마이그레이션 레지스트리에 `M-{이전 MAJOR}-to-{새 MAJOR}` 추가
7. CHANGELOG에 **Breaking Changes** 섹션 작성

### 5.4 CHANGELOG 형식

```markdown
## [1.2.0] — 2026-04-15

### Breaking Changes (MAJOR일 때만)
- 설명

### Added (MINOR)
- 설명

### Changed (MINOR)
- 설명

### Fixed (PATCH)
- 설명
```

## 6. 과거 버전과의 관계

| 레거시 버전 | 의미 | semver 매핑 |
|------------|------|------------|
| v1 ~ v5.2 | 스킬 진화 이력 (비공식) | 참조 전용, 호환 고려 불필요 |
| "3.3" (harness.version) | pre-semver 하네스 스키마 | 부트스트랩 마이그레이션으로 1.0.0 편입 |

`references/project-context.md` § 4의 기존 버전 히스토리(v1~v5.2)는 **참고 기록으로 유지**하되, 1.0.0부터 semver 히스토리를 새로 시작한다.

## 7. 예시: 최근 변경에 semver 적용해보기

| 세션 | 변경 | semver 수준 | 근거 |
|------|------|------------|------|
| Session 14 | harness-feedback 구현 | MINOR | 새 기능 (이슈 보고 프로세스) |
| Session 14 | HARNESS_FRICTION.md 카테고리 추가 | MINOR | 새 기능 (템플릿 확장) |
| Session 15 | `allowed-tools` 누락 수정 | PATCH | 버그 수정 (스킬 로딩 실패) |
| Session 16 | scaffold 심볼릭 링크 수정 | PATCH | 버그 수정 (디스커버리 실패) |
| Session 17 | Plan 모드 TDD 브릿지 | MINOR | 새 기능 (session-routine.md에 섹션 추가) |
